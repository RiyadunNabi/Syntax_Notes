# Git Migration: Windows → Linux (Pop!_OS)
### A Personal Reference by Riyadun Nabi | June 2026

---

## The Situation

I had a folder `BUET_CSE` on Windows 11 with all my semester codes. Inside it:
- The main `BUET_CSE` repo (tracked by GitHub)
- Three inner project repos (each with their own GitHub remote), kept in `.gitignore` of the main repo:
  - `Pong_Game_1-1` → `1-1/MyProject/`
  - `CricketPlayerDatabseSystem_1-2` → `1-2/CSE 108/Project/MyJavaFXProject/`
  - `Pulse-Point-Doctor-Patient-Management-System` → `2-1/Pulse-Point-Doctor-Patient-Management-System/`

I had **uncommitted changes** on Windows before zipping and moving to Pop!_OS.

---

## Step 1: Zip vs Clone — Which to Use?

### The Question
When setting up on a new machine, should I clone from GitHub or unzip from Google Drive?

### The Answer
**Always check if you had uncommitted changes first.**

```bash
# On the old machine (or in the zip), run:
git status
```

If you see:
```
modified: some/file.cpp
no changes added to commit
```
→ **Use the zip from Drive.** Cloning from GitHub would lose those uncommitted changes.

If everything was committed and pushed:
→ **Clone from GitHub.** Cleaner and no zip needed.

### In My Case
I had hundreds of uncommitted files (the `git status` showed `no changes added to commit`), so I used the zip.

## Step 3: Re-link Local Repos to GitHub Remotes

When you move to a new machine, git doesn't automatically know where to push. You need to re-point each repo to its remote.

```bash
# Syntax: git remote set-url origin <github-url>

# Main repo
cd ~/CSE_Study/BUET_CSE
git remote set-url origin https://github.com/RiyadunNabi/BUET_CSE.git

# Pulse Point
cd ~/CSE_Study/BUET_CSE/"2-1/Pulse-Point-Doctor-Patient-Management-System"
git remote set-url origin https://github.com/RiyadunNabi/Pulse-Point-Doctor-Patient-Management-System.git

# Cricket Player DB
cd ~/CSE_Study/BUET_CSE/"1-2/CSE 108/Project/MyJavaFXProject"
git remote set-url origin https://github.com/RiyadunNabi/CricketPlayerDatabseSystem_1-2.git

# Pong
cd ~/CSE_Study/BUET_CSE/"1-1/MyProject"
git remote set-url origin https://github.com/RiyadunNabi/Pong_Game_1-1.git
```

### Why `set-url` and not `remote add`?
Because the `.git` folder from Windows already had a remote configured. `set-url` updates the existing one. `remote add` would throw an error saying "remote origin already exists."

## Step 4: Commit and Push the Uncommitted Changes

```bash
# For each repo:
git add .
git commit -m "sync uncommitted changes from windows"
git push origin main   # or 'master' depending on the repo
```

### Branch Name Issue
Pong used `master` (older Git default) instead of `main`. Pushing to `main` failed:
```
error: src refspec main does not match any
```
Fix: just use the correct branch name:
```bash
git push origin master
```

> **Lesson:** Always check your branch name with `git branch` if push fails.

## Step 5: The CRLF Problem

### What Happened
After pushing, GitHub showed every single file as "modified 4 minutes ago" — even files I hadn't touched in months. The commit showed:
```
380 files changed, 48152 insertions(+), 48152 deletions(-)
```

### Why
- **Windows** saves files with `CRLF` line endings (`\r\n` — carriage return + newline)
- **Linux** uses `LF` only (`\n` — just newline)
- Git saw every single line in every file as "changed" because the invisible `\r` character was different

This is called the **CRLF problem**. It's pure noise — your actual code is identical.

### How to Verify It Was Just CRLF (Not Real Changes)
```bash
git diff HEAD~1 HEAD --stat
```
The signature of a CRLF-only commit is **equal insertions and deletions**:
```
52136 insertions(+), 52136 deletions(-)
```
If real code had changed, these numbers would be unequal.

You can also inspect a specific file:
```bash
git diff HEAD~1 HEAD -- "some/file.cpp"
```

## Step 6: Fix Line Endings Permanently

### The Fix
Add a `.gitattributes` file to each repo. This tells git how to handle line endings — for everyone, on every OS.

```bash
# Run this in each repo root:
git config core.autocrlf input
echo "* text=auto eol=lf" > .gitattributes
git add .gitattributes
git rm --cached -r .        # remove everything from git's index (not from disk!)
git add .                   # re-add everything with normalized line endings
git commit -m "normalize line endings to LF"
git push origin main --force
```

### What Each Command Does

| Command | What it does |
|---|---|
| `git config core.autocrlf input` | On this machine: convert CRLF→LF when committing, don't touch on checkout |
| `echo "* text=auto eol=lf" > .gitattributes` | Create the file that tells git: always store LF in the repo |
| `git add .gitattributes` | Stage the new file |
| `git rm --cached -r .` | Clear git's index completely (files stay on disk!) |
| `git add .` | Re-add everything — now with LF endings |
| `git commit -m "..."` | Commit the normalized version |
| `git push --force` | Overwrite the noisy commit on GitHub |

### What `.gitattributes` Does
```
* text=auto eol=lf
```
- `*` → applies to all files
- `text=auto` → git auto-detects which files are text vs binary
- `eol=lf` → always store LF in the repo

This lives inside the repo itself, so it applies to **anyone who clones** it, on any OS.

## Step 7: The Timestamp Lesson (Painful But Important)

### What I Lost
After force pushing, all files on GitHub showed "committed today." The original timestamps were gone.

### Why It Was Already Too Late
The timestamps were lost at **Step 4**, not Step 6. Here's why:

The files were **never committed on Windows** — they sat as uncommitted local changes. So when I did the first `git commit` on Pop!_OS, git recorded **today's date** as the commit time. There was no previous commit to restore.

Force pushing in Step 6 only made it worse (two noisy commits → one clean one), but the original "timestamp" never existed in git to begin with.

### The Real Lesson
> **Always commit before switching machines.** Even a simple `git commit -m "wip"` preserves history.

Before zipping and moving anywhere, run in every repo:
```bash
git add .
git commit -m "save before moving to new machine"
git push
```

---

## Future-Proofing: Do This on Every New Machine

```bash
# Set globally so all repos on this machine behave correctly
git config --global core.autocrlf input
```

And always create `.gitattributes` on the **first commit** of any new repo:
```bash
echo "* text=auto eol=lf" > .gitattributes
git add .gitattributes
git commit -m "add gitattributes for line ending normalization"
```

## Quick Reference: OS Compatibility

| Setting | Windows | Linux/Mac |
|---|---|---|
| Default line ending | CRLF (`\r\n`) | LF (`\n`) |
| `core.autocrlf input` | CRLF→LF on commit | LF stays LF |
| `.gitattributes eol=lf` | Repo always stores LF | Repo always stores LF |

With `.gitattributes` committed in the repo, line endings stay correct **regardless of OS** — Windows, Linux, or Mac.
