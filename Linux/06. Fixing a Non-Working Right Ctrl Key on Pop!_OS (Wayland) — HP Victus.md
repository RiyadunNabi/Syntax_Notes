# Fixing a Non-Working Right Ctrl Key on Pop!_OS (Wayland) — HP Victus

## The Problem

On some HP Victus laptops, the right Ctrl key (next to the spacebar, left of the arrow keys)
worked fine on Windows but does nothing — or types weird Compose-key characters — on Pop!_OS.

This happens because:
- The key is wired by HP as keycode `105`, but Linux maps that keycode to `Multi_key` (Compose key) instead of `Control_R`.
- Pop!_OS uses **Wayland** by default, where the old `xmodmap` tool (which works on X11) is ignored or reset automatically. So even if you "fix" it with xmodmap, it won't stick.

## The Solution: `keyd`

`keyd` is a kernel-level key remapping daemon. It works on both X11 and Wayland because it
intercepts key presses before the display server even sees them.

### Step 1 — Install build tools and clone keyd

```bash
sudo apt update
sudo apt install git build-essential -y
git clone https://github.com/rvaiya/keyd
cd keyd
```

### Step 2 — Build and install

```bash
make
sudo make install
sudo systemctl enable --now keyd
```

This compiles keyd and starts it as a background service that runs automatically on boot.

### Step 3 — Create the config file

```bash
sudo mkdir -p /etc/keyd
sudo nano /etc/keyd/default.conf
```

Paste this into the file:

```ini
[ids]
*

[main]
rightcontrol = leftcontrol
```

- `[ids] *` → apply this rule to all keyboards.
- `[main] rightcontrol = leftcontrol` → whenever the right Ctrl key is pressed, treat it exactly like left Ctrl.

Save and exit (in `nano`: press `Ctrl+O`, then `Enter`, then `Ctrl+X`).

### Step 4 — Apply the changes

```bash
sudo keyd reload
```

No reboot or logout needed. Test the right Ctrl key immediately (e.g. select text and press it + `C` to copy).

## Verifying It Worked

You can check keyd's status anytime with:

```bash
sudo systemctl status keyd
```

It should show `active (running)`.

## Reinstalling After a Fresh OS Install

`keyd` is built from source here, not installed via `apt`, so it won't survive a Pop!_OS
reinstall or reset. If that happens, just repeat Steps 1–4 above.

## Troubleshooting Notes (for reference)

- `xev | grep keycode` is a useful tool to see what keycode/keysym a key sends.
- On Wayland, `xmodmap` changes may *appear* to apply (xev shows the new keysym) but the
  modifier state (`state 0x0` instead of `state 0x4`) won't actually register as Ctrl —
  that's the telltale sign you need a kernel-level fix like `keyd` instead.
- Other remapping options that exist for X11 (`.Xmodmap`, GNOME autostart scripts) don't
  reliably work on Wayland sessions.

## Other Useful keyd Remaps (Optional)

You can remap other keys the same way. Some examples:

```ini
[main]
capslock = esc          # Caps Lock acts as Escape
leftalt = leftmeta       # Left Alt acts as Super/Windows key
rightcontrol = leftcontrol
```

Edit `/etc/keyd/default.conf`, then run `sudo keyd reload` after any change.
